# Owner Feedback Lessons

This file is a hard gate for 5fedu projects when work touches database, auth, employee, migration, seed, services, or employee forms.

## Database

- App table `id` must be `int8` and auto-incrementing.
- Supabase/PostgreSQL supports this with `generated by default as identity` or `bigserial`.
- Do not use `uuid` or `text` as the primary key for normal app tables unless the sheet/owner explicitly confirms the exception.
- Foreign keys to app tables must use `int8`.
- After migration, verify `information_schema.columns` for `data_type = bigint` and identity/default sequence.

## Employee

Employee tables stay lean unless the sheet says otherwise:

- `id`
- `ho_va_ten`
- `avatar`
- `trang_thai`
- `id_phong_ban`
- `id_chuc_vu`
- `so_dien_thoai`
- `email`
- `ten_dang_nhap`

Do not add HR-profile fields such as `ma_nhan_vien`, `ngay_sinh`, `ngay_vao_lam`, `gioi_tinh`, address, contract, bank, tax, insurance, salary, or ID-card fields unless the current project source explicitly confirms them.

## Auth

- Login uses `ten_dang_nhap`, not `ma_nhan_vien`.
- Fake email login maps `admin` to `admin@gmail.com`.
- Default account is `admin` / `5fedu.com`.
- Creating or changing `ten_dang_nhap` must create/update/delete the Supabase Auth user `<ten_dang_nhap>@gmail.com`.
- Default password for employee-created auth accounts is `123456`.
- Supabase service role/admin auth must run only in a server/admin path, never in frontend code.

## Production DDL Gate

- Supabase service role/API key is not enough to claim schema migration was applied.
- Production schema work needs a real DDL channel: Supabase SQL Editor, direct database URL, linked/logged-in Supabase CLI, or a controlled admin SQL RPC.
- If production still has legacy text-id tables or missing `var_*` tables, status is not PASS.
- Back up/export production before running any rebuild migration that drops legacy tables.
