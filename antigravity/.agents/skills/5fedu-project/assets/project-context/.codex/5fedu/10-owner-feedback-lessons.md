# Owner Feedback Lessons

Nguồn: phản hồi của owner/sếp qua ảnh chat người dùng gửi ngày 2026-05-31.

File này là checklist chống lặp lỗi. Khi làm database, auth, bảng nhân viên, migration, seed, service hoặc UI form liên quan, phải đọc file này trước khi sửa. Nếu một thay đổi vi phạm file này thì phải dừng, sửa plan/schema trước, không code tiếp theo cảm tính.

## Luật cứng đã chốt

- `id` của các bảng app phải là `int8` và tự động tăng dần. Supabase/PostgreSQL có hỗ trợ đầy đủ bằng identity/bigserial, không được nói hoặc suy luận rằng Supabase không có tính năng này.
- Cú pháp ưu tiên cho khóa chính mới:

```sql
id int8 generated by default as identity primary key
```

- Cú pháp chấp nhận khi template/migration cũ đã dùng:

```sql
id bigserial primary key
```

- Không dùng `uuid` cho `id` bảng nghiệp vụ/master nếu owner hoặc sheet không chốt rõ.
- Cột foreign key trỏ tới bảng có `id int8` cũng phải dùng `int8`, ví dụ `id_phong_ban int8`, `id_chuc_vu int8`, `id_nhan_vien int8`, `id_tai_xe int8`.
- Chỉ dùng `text` cho key dạng slug thật sự, ví dụ `id_module = 'nhan-vien'` trong bảng phân quyền nếu source đã chốt như vậy.

## Login và Supabase Auth

- Login chuẩn dùng `ten_dang_nhap`, không dùng `ma_nhan_vien`.
- Người dùng nhập `admin` thì app đăng nhập bằng fake email `admin@gmail.com`.
- Tài khoản test mặc định: `admin` / `5fedu.com`.
- Khi tạo nhân viên có `ten_dang_nhap`, phải tạo Supabase Auth user tương ứng với email giả `<ten_dang_nhap>@gmail.com`.
- Khi sửa `ten_dang_nhap`, phải đồng bộ Supabase Auth: đổi email auth từ username cũ sang username mới, hoặc dùng flow admin đã chốt để tạo user mới và xóa user cũ.
- Khi xóa nhân viên hoặc xóa `ten_dang_nhap`, phải xóa hoặc vô hiệu hóa Supabase Auth user tương ứng theo flow admin đã chốt.
- Mật khẩu mặc định cho tài khoản nhân viên do module tạo: `123456`.
- Service role/admin auth tuyệt đối không đưa vào frontend. Flow này phải chạy qua server/admin path như Edge Function, API server, hoặc script admin có kiểm soát.
- Không tự thêm cột `auth_user_id` vào `var_nhan_vien` nếu sheet/source chưa chốt. Mặc định có thể đối chiếu qua fake email `<ten_dang_nhap>@gmail.com`; nếu cần lưu `auth_user_id`, phải ghi rõ lý do và xin chốt.

## Checklist bắt buộc trước migration/schema

Trước khi viết hoặc sửa SQL migration thật:

1. Đối chiếu từng bảng với sheet/source chính.
2. Lập bảng kiểm: table, primary key type, identity/auto increment, foreign key type, extra columns.
3. Xóa hoặc đánh dấu hỏi lại mọi trường không có trong source, đặc biệt ở `var_nhan_vien`.
4. Kiểm tra tất cả `id` bảng app là `int8` auto increment.
5. Kiểm tra foreign key tới bảng app là `int8`.
6. Kiểm tra login dùng `ten_dang_nhap`, không dùng `ma_nhan_vien`.
7. Với auth/admin flow, lập locked plan trước khi sửa code hoặc database.

## Verify bắt buộc

Sau migration hoặc schema change, phải verify bằng SQL không in secret:

```sql
select table_name, column_name, data_type, is_identity, column_default
from information_schema.columns
where table_schema = 'public'
  and column_name = 'id'
order by table_name;
```

Kết quả đạt yêu cầu khi mỗi bảng app có `data_type = bigint` và `is_identity = YES` hoặc `column_default` có `nextval(...)`.

Với bảng nhân viên, verify danh sách cột:

```sql
select column_name, data_type
from information_schema.columns
where table_schema = 'public'
  and table_name = 'var_nhan_vien'
order by ordinal_position;
```

Kết quả không được có trường hồ sơ nhân sự mở rộng nếu source chưa chốt.

Với auth, smoke test tối thiểu:

- Login `admin` / `5fedu.com` thành công.
- Tạo nhân viên có `ten_dang_nhap` tạo được Supabase Auth user `<ten_dang_nhap>@gmail.com`.
- Đổi `ten_dang_nhap` đồng bộ Auth đúng.
- Xóa nhân viên hoặc xóa username xử lý Auth user đúng.

## Cách phản ứng khi phát hiện lệch

- Nếu migration/code hiện tại dùng `uuid` cho `id`: dừng mở rộng, tạo plan sửa schema.
- Nếu bảng nhân viên đang nhiều trường hơn rule đã chốt: dừng mở rộng module nhân viên, tạo plan cleanup.
- Nếu login đang dựa vào mã nhân viên: dừng, sửa auth model sang `ten_dang_nhap`.
- Nếu chưa đọc được Google Sheet do thiếu auth: mở browser để người dùng đăng nhập Google hoặc xin quyền truy cập sheet trước khi suy luận tiếp.

## Production DDL Gate

- Supabase service role/API key chỉ đủ cho REST/DML/Auth admin; không được giả định nó chạy được DDL/migration.
- Trước khi nói đã apply production schema, phải có một trong các kênh DDL thật: Supabase SQL Editor, direct database URL, Supabase CLI đã link/login, hoặc RPC admin `exec_sql` đã được tạo có kiểm soát.
- Nếu production check thấy bảng legacy text-id vẫn còn, trạng thái không được báo PASS. Phải ghi rõ bảng thiếu/sai và giữ task ở PARTIAL/BLOCKED cho đến khi migration rebuild được chạy.
- Với production đang có dữ liệu sai schema, phải backup/export trước khi chạy migration có `drop table ... cascade`.

Các lỗi mới đã chốt không được lặp lại:

- Không dùng một `TransportModulePage`/config chung nếu form/detail/action/tổng hợp của từng module cần nghiệp vụ riêng.

- Không đặt nút duyệt trong form. Duyệt là action riêng.

- Không để detail tài xế/địa điểm/xe chỉ hiển thị field thô nếu nghiệp vụ cần lịch sử liên quan.
